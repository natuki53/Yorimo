# 12. Future Extension

このドキュメントでは、YorimoをMVPからプロダクト品質へ拡張するための計画を整理します。ここに書く内容は今後の想定であり、現在の実装済み機能とは分けて扱います。

## Google Maps API連携

目的:

- 実在スポットの検索
- 地図表示
- Place IDによるスポット正規化
- 営業時間、写真、評価、住所の補完

追加案:

- `Spot.externalPlaceId`
- `Spot.provider`
- `Spot.openingHoursJson`
- `Spot.lastSyncedAt`

注意点:

- Google Maps APIの利用規約に沿ったキャッシュ設計
- 料金上限とAPI quota管理
- ユーザー生成スポットと外部スポットの重複排除

## 乗換・徒歩ルート連携

目的:

- 通学、通勤ルート上で本当に寄りやすい場所を推薦する
- 現在の直線距離ではなく、徒歩時間や乗換時間で評価する

拡張案:

- 経由駅だけでなく経路segmentを保存
- `RouteSegment` テーブル追加
- 徒歩時間、乗車時間、寄り道による追加時間をスコア化
- Google Directions APIや公共交通APIとの連携

## AI推薦

目的:

- ユーザーの好み、投稿文、写真、時間帯、天気、混雑などをより柔軟に扱う
- ルールだけでは表現しにくい「気分に合う」推薦を強化する

段階的な移行:

1. 現在のルールベーススコアを維持
2. `RecommendationLog` で表示、クリック、保存、訪問、skipを記録
3. SpotとPostをembedding化
4. ベクトル検索で候補抽出
5. AIスコアとルールベーススコアを合成
6. `reasons` は安全なテンプレートで説明可能性を保つ

## ベクトル検索

対象データ:

- `Spot.name`
- `Spot.description`
- `Spot.category`
- `Spot.tags`
- 投稿の `caption`
- ユーザーの興味タグ

構成案:

- PostgreSQL + pgvector
- または外部ベクトルDB
- `SpotEmbedding` テーブル
- embedding更新ジョブ

ユースケース:

- 「静かに作業したい」
- 「雨の日に友達と行ける場所」
- 「30分だけ甘いもの」

## 投稿画像・動画アップロード

現在は `mediaUrl` を受け取るだけです。今後はアップロード基盤が必要です。

拡張案:

- `POST /api/media/upload-url`
- 署名付きURLでクライアント直接アップロード
- アップロード完了通知API
- 画像リサイズとサムネイル生成
- 動画の長さ、サイズ、形式制限
- `MediaAsset` テーブル追加

## S3 / Cloudinary連携

`mediaService` を以下の責務に拡張します。

- 署名付きURL発行
- public URLまたはCDN URL生成
- 投稿削除時のファイル削除
- サムネイル、動画変換後URL管理
- 不適切コンテンツ検出との連携

Cloudinaryを使う場合は変換とサムネイル生成を任せやすく、S3を使う場合は柔軟性とコスト管理をしやすいです。

## リアルタイム通知

対象:

- 投稿へのlike
- コメント
- フォロー
- 通報対応結果
- おすすめスポット更新

実装案:

- `Notification` テーブル
- WebSocketまたはServer-Sent Events
- Push通知
- 通知既読管理

高校生も対象に含むため、通知頻度や夜間通知制限も検討が必要です。

## フォロー機能

`Visibility.followers` を正しく活かすにはフォロー機能が必要です。

追加案:

- `Follow` テーブル
- `followRequest` または承認制follow
- `followers` 投稿の閲覧条件
- フィードでフォロー中ユーザーを優先表示
- ブロック時のフォロー解除

## レコメンド精度改善

改善案:

- 保存、訪問、skip、dislikeの重みをA/Bテストで調整
- 時間帯、曜日、天気、混雑度を追加
- spotの営業中判定
- 年齢帯に合わないカテゴリの除外
- 同じスポットばかり出さない多様性制御
- 新規ユーザー向けの人気スポットfallback
- 説明可能な `scoreBreakdown` を画面で見せる

## 管理画面

必要な機能:

- 通報一覧
- 投稿非表示
- ユーザー停止
- スポット審査
- タグ管理
- NGワード管理
- 利用状況ダッシュボード

バックエンド拡張:

- `Role`
- `AdminActionLog`
- `ModerationCase`
- 管理者専用middleware

## モデレーション機能

優先度が高い理由:

- 投稿型SNSである
- 位置情報に近い情報を扱う
- 高校生ユーザーを想定している

実装案:

- 通報件数の閾値で一時非表示
- AI画像・テキスト検査
- 管理者レビュー
- ユーザーへの通知
- 異議申し立て

## コンテスト発表に向けた見せ方

伝えるべき独自性:

- 「目的地検索」ではなく「日常ルートの途中にある寄り道」を推薦する
- `currentLat/currentLng`、`routeId`、`availableMinutes`、`budget`、`mood` を同時に扱う
- `yorimichiScore` を7要素に分解し、推薦理由を説明できる
- 地図上のスポットとSNS投稿が同じデータモデルでつながる
- 安全設計として通報、ブロック、公開範囲、現在地非保存を初期から考慮している

デモで見せる流れ:

1. ユーザー登録
2. 興味タグとデフォルト予算を設定
3. 学校帰りルートを登録
4. 現在地、空き時間、気分を入力して推薦取得
5. `yorimichiScore` と `reasons` を表示
6. スポット詳細で投稿を表示
7. 訪問後にレビュー投稿
8. 保存やフィードバックが次回推薦に反映されることを説明

## 優先ロードマップ

| 優先度 | 拡張 | 理由 |
| --- | --- | --- |
| 高 | フロントエンド実装 | API価値を体験として見せるため |
| 高 | media upload | SNS投稿らしさを出すため |
| 高 | フォロー機能 | `followers` visibilityを成立させるため |
| 高 | モデレーション管理 | 高校生を含むSNSとして必要 |
| 中 | Google Maps API | 実在スポットと地図UXを強化 |
| 中 | AI推薦 | 独自性と精度を高める |
| 中 | RecommendationLog | 推薦改善のための学習データ |
| 低 | リアルタイム通知 | SNS継続利用を高める |
